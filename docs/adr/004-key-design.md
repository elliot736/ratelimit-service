# ADR-004: Key Design and Namespacing

## Status

Accepted

## Date

2025-10-20

## Context

Rate limit keys in Redis must satisfy several requirements:

1. **Uniqueness:** Different clients must get different keys. Two users hitting the same endpoint should not share a rate limit counter.
2. **Policy isolation:** Different rate limit policies (e.g., per-second vs. per-hour) must use different keys, even for the same client.
3. **Collision avoidance:** Multiple applications or services sharing the same Redis instance must not interfere with each other's rate limit keys.
4. **Debuggability:** An operator should be able to inspect Redis with `SCAN` or `KEYS` and understand what a key represents.
5. **Predictability:** The key format should be deterministic — the same client with the same policy always produces the same key.

## Decision

### Key Format

Keys follow the pattern: `{prefix}:{policy}:{identifier}`

- **prefix** — Configurable, defaults to `rl`. Prevents collision with non-rate-limit keys in shared Redis instances.
- **policy** — The policy name (e.g., `api`, `burst`, `hourly`). Ensures different policies for the same client use different counters.
- **identifier** — Generated by the key generator from the request (e.g., `ip:203.0.113.50`, `user:u-abc123`).

Examples:
```
rl:api:ip:203.0.113.50
rl:burst:user:u-abc123
rl:per-second:apikey:sk-live-xyz
rl:hourly:header:X-Tenant-ID:acme
```

### Key Generators

Key generators are pluggable functions with the signature `(req: KeyGeneratorRequest) => string`. Built-in generators:

- **`byIp`** — Uses `x-forwarded-for` header (first value) or `req.ip`. Produces `ip:{address}`.
- **`byUserId`** — Uses `req.userId`. Produces `user:{id}`. Throws if not present.
- **`byApiKey(header?)`** — Uses a configurable header (default: `x-api-key`). Produces `apikey:{value}`.
- **`byHeader(name)`** — Uses any arbitrary header. Produces `header:{name}:{value}`.
- **`composite(...generators)`** — Concatenates multiple generators with `:`. Produces `ip:1.2.3.4:user:u-123`.
- **`custom(fn)`** — Wraps any function for full control.

### Composite Policy Keys

When using `CompositePolicy`, each sub-policy gets its own key: `{prefix}:{sub-policy-name}:{identifier}`. This ensures independent tracking — the per-second counter is separate from the per-hour counter.

## Trade-offs

**Benefits:**
- Predictable structure makes Redis debugging straightforward. `SCAN 0 MATCH rl:api:*` shows all API rate limit keys.
- Namespacing by policy name prevents cross-contamination between different limit windows.
- The configurable prefix allows multiple applications to share a Redis instance safely.

**Costs:**
- Composite keys can get long: `rl:api:ip:203.0.113.50:header:X-Tenant-ID:acme-corp`. Long keys consume more Redis memory and bandwidth. In practice, key overhead is negligible compared to the data stored per key.
- The `:` separator means identifiers containing `:` could cause ambiguity. This is acceptable because the key is used as an opaque string by Redis — it does not need to be parsed back into components.

## Consequences

1. The `RateLimiter` class prepends `{prefix}:` to all keys passed to `consume()` and `reset()`.
2. Middleware adapters construct the full key from the policy name and key generator result.
3. Key generators are pure functions with no side effects, making them easy to test and compose.
4. The `KeyGeneratorRequest` interface provides a normalized view of the request for key generation, decoupled from any specific web framework.
